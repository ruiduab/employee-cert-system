# 联想职工认证产品文档 v2.0

## 一、产品概述

联想职工认证是一套面向企业员工和专业人士的身份认证体系，通过4个步骤的标准化流程，完成员工身份验证并授予相应权益。

---

## 二、认证基础条件（关键前置条件）

用户必须同时满足以下两个条件才能进行职工身份认证：

### 2.1 条件一：用户登录
- 用户必须完成登录
- 登录后系统识别用户身份

### 2.2 条件二：完成个人实名认证
- 用户必须完成个人实名认证（第2步）
- 这是进行任何职工身份认证的必要前置条件
- 系统通过 `hasRealname` 字段跟踪认证状态

**关键说明**：个人实名认证（第2步）是基础性认证，与后续的职工身份认证（第3、4步）分开：
- 个人实名认证：验证个人身份的真实性
- 职工身份认证：验证企业员工身份和职称

---

## 三、认证流程（核心交互逻辑）

### 3.1 流程决策树

```
用户进入系统
    ↓
显示第1步：选择认证类型
    ↓
用户选择认证类型 + 点击"下一步"
    ↓
[系统判断] hasRealname 状态？
    ├─ hasRealname = false（未完成个人实名认证）
    │   ├─ 弹窗提示："你还未完成个人实名认证，需要先完成实名认证才能进行职工认证"
    │   ├─ 用户选择"继续进行实名认证"
    │   └─ → 进入第2步（未完成状态）
    │       ├─ 用户选择认证方式（身份证或人脸识别）
    │       ├─ 点击"下一步"
    │       ├─ 系统标记 hasRealname = true
    │       └─ → 跳过第2步，直接进入第3步
    │
    └─ hasRealname = true（已完成个人实名认证）
        ├─ 无弹窗提示
        ├─ 系统检查专业职称要求（如适用）
        └─ → 直接进入第3步
            └─ 第2步显示"已完成"状态（可返回查看）
```

## 四、详细交互规则

### 4.1 第1步：选择认证类型

**页面显示内容**：
- 两种认证类型选项：
  - 基础职员认证
  - 专业职称认证（需选择具体职业）

**交互流程**：

#### A. 用户尚未完成个人实名认证（hasRealname = false）

**步骤1：选择认证类型**
- 用户在第1步选择认证类型（基础职员或专业职称）
- 如选择专业职称，需进一步选择四个职业之一

**步骤2：点击"下一步"按钮**
- 系统触发前置检查

**步骤3：系统执行检查逻辑**
```
检查流程：
1. 首先检查 hasRealname 状态
2. 如果 hasRealname = false，弹窗提示
3. 如果选择专业职称但未选择具体职业，提示错误
```

**步骤4：弹窗提示**
- **标题**："完成实名认证"
- **内容**："你还未完成个人实名认证，需要先完成实名认证才能进行职工认证。"
- **按钮选项**：
  - ✓ "继续进行实名认证" → 跳转至第2步
  - ✗ "返回" → 留在第1步（用户可修改选择）

**步骤5：用户点击"继续进行实名认证"**
- 弹窗关闭
- 系统跳转至第2步（未完成状态）

#### B. 用户已完成个人实名认证（hasRealname = true）

**步骤1：选择认证类型**
- 用户在第1步选择认证类型

**步骤2：点击"下一步"按钮**
- 系统触发前置检查

**步骤3：系统执行检查逻辑**
```
检查流程：
1. 检查 hasRealname 状态
2. 如果 hasRealname = true，跳过实名认证检查
3. 如果选择专业职称但未选择具体职业，提示错误
4. 所有检查通过，直接进入第3步
```

**步骤4：直接进入第3步**
- 无弹窗提示
- 用户直接看到第3步内容（企业在职认证）
- 第2步已完成状态被跳过，但用户可通过返回按钮查看

---

### 4.2 第2步：个人实名认证

**作用**：验证用户身份的真实性，确保身份信息真实有效

**隐私保护**：
- 身份信息将被加密存储
- 仅用于身份验证，不会泄露给第三方

#### 状态一：未完成状态（首次进入）

**显示内容**：
- 页面标题："第2步：实名认证"
- 说明文字："选择认证方式验证你的身份信息"
- 两种认证方式可选：

**方式1：身份证认证（推荐）**
- 所需信息：真实姓名（2-20个汉字）+ 身份证号码（18位）
- 说明："请输入您的真实身份信息。身份证号码需为18位，请确保信息准确无误。"
- 流程：输入信息 → 系统校验 → 验证通过

**方式2：人脸识别认证**
- 所需信息：真实姓名（2-20个汉字）+ 人脸识别
- 说明："通过人脸识别进行快速身份验证。请在光线充足的环境下进行，确保面部清晰可见。过程中请勿佩戴墨镜、帽子等遮挡物。"
- 流程：输入姓名 → 点击"进行人脸识别" → 识别通过

**交互规则**：
- 用户可选择任意一种方式
- 两种方式都通过后，点击"下一步"

**系统操作**：
```
当用户在第2步点击"下一步"时：
1. 系统标记 hasRealname = true
2. 系统判断用户当前来自哪里：
   - 如果来自第1步的弹窗 → 进入第3步
   - 如果来自其他位置 → 进入第3步
3. 自动跳过第2步，直接显示第3步
```

#### 状态二：已完成状态（返回时）

**显示内容**：
- 页面标题："第2步：实名认证"
- 成功标志：✅
- 提示文案："实名认证已完成"
- 说明："你已完成个人实名认证，可以继续进行职工认证"

**交互规则**：
- 用户可查看已完成状态
- 点击"下一步" → 继续进入第3步
- 点击"返回" → 回到第1步

**关键机制**：
- 第2步一旦标记为已完成（hasRealname = true），无法再次修改
- 用户返回到第2步时，始终显示已完成状态
- 即使返回多次，也不需要重新认证

---

### 4.3 第3步：企业在职认证

**显示触发条件**：
- 用户从第1步完成判断后进入
- 无论用户是否完成个人实名认证，都会进入此步

**所需信息**：
1. 企业名称（必填）- 支持搜索和输入修改
2. 职位信息（必填）- 2-12字符
3. 所属行业（必填）- 下拉选择

**行业和职位匹配规则**：

| 认证类型 | 行业限制 | 职位要求 |
|---------|---------|---------|
| 基础职员认证 | 无限制 | 无限制 |
| 法律专业职称 | 仅"法律" | 律师、法务、顾问、法律顾问等 |
| 工业设计专业职称 | 工业设计、制造业、消费电子 | 工业设计师、产品设计师等 |
| 媒体设计专业职称 | 设计、媒体、广告、创意产业 | UI设计师、平面设计师等 |
| 编程开发专业职称 | 互联网/信息技术、软件开发、云计算 | 工程师、开发者等 |

---

### 4.4 第4步：选择在职证明方式

**说明**：用户可从以下四种方式中任选其一，提交在职身份证明

**方式1：企业邮箱认证（推荐）**
- 说明："使用企业邮箱认证，仅可认证为所属企业员工。如需认证具体职务请上传其他证明材料或选择其他认证方式。"
- 所需：企业邮箱 + 验证码
- 流程：输入邮箱 → 发送验证码 → 输入验证码 → 验证

**方式2：劳动合同或在职证明**
- 说明："请上传劳动合同或企业在职证明。在职证明需为六个月内开具，需包含真实、清晰、完整的关键信息，无遮挡。"
- 所需：文件上传（JPG/PNG/PDF，5MB以内）

**方式3：个人所得税APP认证**
- 说明："请上传个人所得税APP由「个人中心」进入「在职受雇信息」页面的录屏视频。"
- 所需：视频上传

**方式4：其他材料认证**
- 说明："请上传可以证明在职公司、目前职位和在职时效的材料。如社保缴费证明、职称材料等。"
- 所需：文件上传（支持多项组合）

## 五、关键判断逻辑总结

### 5.1 系统状态跟踪

系统通过以下字段跟踪用户状态：
- `hasRealname`：个人实名认证状态（true/false）
- `selectedType`：认证类型（'basic'/'specialist'）
- `selectedSpecialty`：职业分类（仅专业职称适用）
- `currentStep`：当前步骤（1-5）

### 5.2 第1步的判断逻辑

```
用户点击"下一步"时：

判断1：检查 hasRealname
├─ false → 显示弹窗
├─ true → 继续进行判断2

判断2：检查认证类型（仅当 hasRealname = true 时）
├─ specialist 且 selectedSpecialty = 空 → 提示错误
├─ 其他情况 → 进入第3步
```

### 5.3 第2步的判断逻辑

```
用户点击"下一步"时：

步骤1：检查当前状态
├─ 如果 hasRealname = false → 标记为 true
├─ 如果 hasRealname = true → 保持不变

步骤2：执行流程跳转
├─ currentStep = 3（直接进入第3步）
```

### 5.4 返回按钮逻辑

```
用户点击"返回"时：

currentStep 减 1，进入前一步
- 从第3步返回 → 进入第2步（显示已完成状态）
- 从第2步返回 → 进入第1步
- 从第1步返回 → 无操作（已在第1步）
```

## 六、用户交互流程演示

### 场景一：首次使用（未完成实名认证）

```
1. 用户进入第1步
   - hasRealname = false
   - 选择认证类型（如：基础职员认证）
   
2. 点击"下一步"
   - 系统检查发现 hasRealname = false
   - 显示弹窗提示
   
3. 点击"继续进行实名认证"
   - 进入第2步（未完成状态）
   - 显示两种认证方式
   
4. 选择认证方式，填写信息
   - 例：身份证认证，输入姓名和身份证号
   
5. 点击"下一步"
   - 系统标记 hasRealname = true
   - 自动跳过第2步
   - 进入第3步（企业在职认证）
   
6. 填写企业信息，继续认证流程
   - 第3步 → 第4步 → 成功页面
```

### 场景二：已完成实名认证

```
1. 用户进入第1步
   - hasRealname = true（系统已记录）
   - 选择认证类型
   
2. 点击"下一步"
   - 系统检查发现 hasRealname = true
   - 无弹窗提示
   - 直接进入第3步
   
3. 用户注意到被跳过了第2步
   - 可点击"返回"回到第2步查看已完成状态
   
4. 查看第2步的已完成状态
   - 显示"✅ 实名认证已完成"
   
5. 可再次点击"下一步"
   - 重新进入第3步
```

### 场景三：修改认证类型（已完成实名认证）

```
1. 用户已完成实名认证，在第1步修改认证类型
   - 从基础职员认证改为专业职称认证
   - 需选择具体职业（如：编程开发）
   
2. 点击"下一步"
   - 系统检查1：hasRealname = true ✓
   - 系统检查2：selectedSpecialty = 已选择 ✓
   - 所有检查通过
   - 直接进入第3步
```

## 七、验证和错误提示

### 7.1 第1步错误提示

| 错误情况 | 提示信息 | 处理方案 |
|---------|--------|--------|
| 未完成实名认证 | "你还未完成个人实名认证，需要先完成实名认证才能进行职工认证" | 弹窗 + 引导至第2步 |
| 选择专业职称但未选职业 | "请选择你的具体职业" | 提示框 + 留在第1步 |

### 7.2 第3步验证规则

| 字段 | 规则 | 错误提示 |
|------|------|--------|
| 企业名称 | 必填，不能为空 | "请输入企业名称" |
| 职位信息 | 2-12字符，专业职称需匹配职业列表 | "职位长度2-12字符" / "职位不符合要求" |
| 所属行业 | 必填，专业职称需严格匹配 | "请选择行业" / "行业不符合职业要求" |

## 八、系统状态管理

### 8.1 状态持久化

系统在用户会话期间保持以下状态不变：
- 一旦 `hasRealname = true`，用户不需要重复进行实名认证
- 用户选择的认证类型和职业信息保持，直到用户修改

### 8.2 状态恢复

用户返回前一步骤时：
- 所有已填信息保持不变
- 所有已选选项保持不变
- 不需要重新输入或选择

## 九、关键交互点总结

| 交互点 | 判断条件 | 触发动作 | 用户看到的结果 |
|------|--------|--------|-------------|
| 第1步点下一步 | hasRealname = false | 显示弹窗 | 弹窗提示 |
| 弹窗继续认证 | 点击按钮 | 跳转第2步 | 进入实名认证 |
| 第2步点下一步 | 完成认证 | 标记hasRealname = true | 直接进第3步 |
| 第1步返回后再点下一步 | hasRealname = true | 跳过第2步 | 直接进第3步 |
| 第3步点返回 | 返回按钮 | currentStep - 1 | 返回第2步（已完成） |

## 十、流程总结

```
认证基础条件检查
    ↓
[系统判断] 是否完成个人实名认证？
    ├─ 否 → 进行第2步 → 完成后进第3步
    └─ 是 → 直接进第3步（第2步可查看已完成状态）
    
所有用户（无论认证路径）都会经过：
第1步（选择认证类型）→ 第3步（企业认证）→ 第4步（在职证明）→ 成功页面
```
